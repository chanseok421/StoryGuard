Role별 조사와 실행 과제

각 role은 아래 4칸을 채워 오면 됩니다. “찾아야 할 것”은 조사, “해야 할 것”은 오늘 구현/문서 산출물입니다.

Role 1

Frontend / UI / Graph

브라우저에서 실제로 작가가 보는 경험과 필요한 데이터 필드를 정의합니다.

찾아야 할 것

  • 배포 웹에서 localStorage/IndexedDB 사용 가능 범위
  • 긴 원고 저장 시 localStorage 용량 한계
  • Story Graph 라이브러리 후보: SVG, D3, react-force-graph, Cytoscape
  • 카드 클릭 → 노드 강조에 필요한 필드

결정해야 할 것

  • 최근 입력/결과 저장을 localStorage로 할지 IndexedDB로 할지
  • 그래프를 직접 SVG로 할지 라이브러리를 쓸지
  • nodes/edges 최소 필드

해야 할 것

  • mock JSON으로 충돌 카드 UI 먼저 완성
  • nodes/edges 샘플로 주인공 중심 그래프 렌더링
  • 카드 클릭 시 relatedNodeIds 강조
산출물

UI 필요 필드 목록, graph schema 요구사항, 저장 UX 결정표.

Role 2

Backend / API / Supabase

로그인, 개인 데이터 분리, API 계약, fallback 흐름을 책임집니다.

찾아야 할 것

  • Supabase Auth로 회원별 user_id를 얻는 방식
  • RLS에서 auth.uid() = user_id 정책 적용 방법
  • pgvector table에 user_id/project_id 필터를 거는 방법
  • 배포 환경에서 Ollama 사용 불가 시 provider fallback

결정해야 할 것

  • 이번 MVP에 로그인/Supabase를 넣을지, v1로 미룰지
  • 서버에 원고를 영구 저장할지 여부
  • API response schema 최종안

해야 할 것

  • /api/analyze mock response부터 구현
  • Groq → mock fallback 구조 설계
  • Supabase 사용 시 projects/story_chunks 테이블 초안 작성
산출물

저장 정책, RLS 정책 초안, API schema, provider fallback 흐름도.

Role 3

LangChain / RAG / Evidence

설정집을 검색 가능한 기억으로 만들고, 원고와 관련된 근거를 찾아냅니다.

찾아야 할 것

  • localStorage, IndexedDB, in-memory, Supabase pgvector 비교
  • 한국어 embedding 후보: BGE-M3, multilingual-e5, OpenAI embedding 등
  • chunking 단위별 검색 품질 차이
  • LangChain retriever를 Next.js/API에서 붙이는 방식

결정해야 할 것

  • MVP retrieval을 keyword로 시작할지 embedding으로 갈지
  • chunk를 설정 항목 단위로 할지 문단 단위로 할지
  • top-k 개수와 threshold

해야 할 것

  • 샘플 설정 10개 + 테스트 원고 5개 준비
  • 정답 설정이 top-3에 들어오는지 비교
  • relatedSettings/evidence JSON 반환 함수 만들기
산출물

embedding 비교표, chunking 결정안, top-k retrieval 함수, evidence schema.

Role 4

Prompt / LangGraph / Demo

AI가 어떤 기준으로 설정 붕괴를 판단할지 정의하고, 발표 성공률을 끌어올립니다.

찾아야 할 것

  • Groq 모델별 한국어/JSON 안정성
  • Ollama 로컬 모델 후보의 충돌 탐지 품질
  • 설정 붕괴 유형 분류 기준
  • LangGraph 노드를 최소 몇 개로 나눌지

결정해야 할 것

  • 배포 데모에 쓸 생성 모델
  • 로컬 QA에 쓸 모델
  • issue.type과 severity 기준
  • mock fallback 결과 내용

해야 할 것

  • 샘플 작품/설정/원고 작성
  • 의도된 오류 3개 심기
  • LangGraph 노드 초안 작성
  • mock-result.json 완성
산출물

모델 비교표, prompt 초안, LangGraph 노드 목록, 샘플 원고와 mock JSON.

첫 회의 진행 순서

회의가 길어지지 않게 아래 순서대로 결정하세요. 스키마가 먼저 정해져야 4명이 병렬 작업할 수 있습니다.

1. Backend response JSON schema 초안 제시
2. Frontend 화면과 그래프에 필요한 필드 추가 요청
3. RAG relatedSettings/evidence 필드 확정
4. Prompt issues/type/severity 필드 확정
5. 전원 mock-result.json 하나에 합의하고 각자 구현 시작

팀장 체크포인트

오늘의 핵심은 “좋은 조사”가 아니라 “결정 가능한 조사”입니다. 각 담당자는 비교표, 샘플 테스트 결과, schema 초안 중 하나를 반드시 들고 와야 합니다. 말로만 좋은 기술은 보류하고, 샘플 원고 하나를 끝까지 통과시키는 선택을 우선하세요.
